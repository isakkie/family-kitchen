# 架构方向

## 当前原则

家庭厨房分成两层：

- 应用框架：菜谱导入、编辑、搜索、冰箱、推荐、采购清单和菜单规划。可以公开发布和复用。
- 个人数据：菜谱库、冰箱库存、采购清单和菜单安排。默认只保存在本地浏览器，不进入代码仓库。
- 图片数据：当前保存图片 URL。后续可以扩展成本地图片上传、对象存储或自建图床。

## 本地存储

当前版本使用浏览器 `localStorage` 保存数据。每台设备都有自己的数据副本。

适合：

- 个人或家庭早期试用。
- 不想维护服务器。
- 通过 JSON 导出/导入手动同步。

限制：

- 换设备不会自动同步。
- 清空浏览器数据会删除本地菜谱。
- 多人协作需要手动导出和导入。

## 可选云端

后续可以增加一个同步适配层，例如：

```js
const storageAdapter = {
  async load() {},
  async save(state) {}
};
```

默认适配器仍然使用本地浏览器。需要云端的人可以自己接：

- Supabase
- Firebase
- 自建 API
- 私有 Git 仓库中的 JSON 文件
- 家庭 NAS 或其他私有存储

这样框架可以公开，数据仍由使用者自己掌控。

## 不做的事

- 不把个人菜谱 JSON 提交进公开仓库。
- 不把下厨房账号信息、cookie 或验证码保存到项目里。
- 不把前端密码当作真正安全边界。
